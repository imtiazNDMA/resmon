# Frozen inter-pipeline contracts: `Observation` and `AnalyticalBaseTable`

These two schemas are the seams between the three pipelines (see [ADR-0003](../adr/0003-frozen-pipeline-contracts.md)). Remote-Sensing **emits** `Observation`; Data-Engineering **consumes** `Observation` + bulletins and **emits** the ABT; ML **consumes** the ABT. Code all three tracks against these; do not change a column without bumping `contract_version` and notifying the other tracks.

`contract_version: 3`

**Changelog**
- **v3** — Added `evaporation` (mm/day, ERA5-Land open-water evaporation) to the ABT catchment-forcing block as the mass-balance outflow term (§8.3, decision D6). The physical `catchment_forcing` table mirrors it.
- **v2** — Forecast forcing moved out of the wide ABT into a separate horizon-keyed `ForecastForcing` block (§3), because forecast forcing is inherently `(date × horizon)` and does not fit one row per `(reservoir_id, date)`. GFS forecast features are point-in-time reforecasts sourced from GEE `NOAA/GFS0P25` (archive back to 2015 covers the training window), so train and serve build them identically.
- **v1** — Initial freeze of `Observation` and `AnalyticalBaseTable`.

## Conventions

- **Canonical date is the IST calendar date.** Bulletins are IST; SAR and forcing are UTC — both are mapped to the IST date they fall on before joining (FR-ABT-2). Column type `date` everywhere means this.
- **Point-in-time:** every column holds only what was knowable at `date` (FR-ABT-3). Forecast-forcing columns hold the value from the GFS run *issued on or before* `date`, never a later reanalysis.
- **Units are explicit in every column** and never change.
- `NULL` = not available for that `(reservoir_id, date)`; never a silent zero.

## 1. `Observation` (Remote-Sensing → Data-Engineering)

One row per SAR acquisition over a reservoir AOI.

| Column | Type | Unit | Null? | Notes |
| --- | --- | --- | --- | --- |
| `reservoir_id` | text | — | no | FK to Reservoir |
| `acquisition_date` | date (IST) | — | no | PK with `reservoir_id` |
| `surface_area` | float | km² | no | true-area measure (`ee.Image.pixelArea`), never pixel counts (FR-RS-3) |
| `area_confidence` | float | 0–1 | no | cluster separability / mask compactness / layover fraction (FR-RS-3) |
| `derived_volume` | float | BCM | yes | via rating curve; NULL until a curve exists |
| `derived_level` | float | m | yes | via rating curve; NULL until a curve exists |
| `water_mask_ref` | text | — | no | URI/key to the persisted binary mask |
| `extraction_method` | text | — | no | e.g. `kmeans`, `otsu`, `unet` (FR-RS-2) |
| `extraction_version` | text | — | no | version of the pluggable extractor |
| `scene_ids` | text[] | — | no | Sentinel-1 scene IDs in this acquisition (provenance, FR-RS-6) |
| `orbit_relative` | int | — | no | fixed relative orbit per reservoir (FR-RS-1) |
| `pass_direction` | text | — | no | `ASC` or `DESC`, fixed per reservoir (FR-RS-1) |
| `aoi_version` | text | — | no | AOI polygon version used (FR-RS-6) |
| `layover_shadow_fraction` | float | 0–1 | no | fraction of AOI masked for layover/shadow |
| `processing_params` | jsonb | — | no | calibration/speckle/terrain-flattening params (FR-RS-6) |

**Stub rule (so DE/ML start before real SAR lands):** synthetic `Observation` rows may be generated by inverting a rough bulletin-derived rating curve to produce `surface_area`; they carry `extraction_method = 'stub'` so they are trivially filterable.

## 2. `AnalyticalBaseTable` (Data-Engineering → ML)

**Grain:** one row per `(reservoir_id, date)` on a **continuous daily IST calendar**. Daily because forcing is native-daily and the forecast horizon is in days; ground-truth and SAR columns are populated only on the dates they exist and `NULL` otherwise (with `days_since_bulletin` / `days_since_acquisition` carrying recency). Training rows for the forecaster are the dates where a storage target exists.

### Keys & alignment

| Column | Type | Unit | Null? | Notes |
| --- | --- | --- | --- | --- |
| `reservoir_id` | text | — | no | PK with `date` |
| `date` | date (IST) | — | no | continuous daily grid |
| `days_since_bulletin` | int | days | no | recency of the latest ground-truth row |
| `days_since_acquisition` | int | days | no | recency of the latest SAR `Observation` |

### Ground truth (from bulletins; populated on bulletin dates)

| Column | Type | Unit | Null? |
| --- | --- | --- | --- |
| `gt_level` | float | m | yes |
| `gt_live_storage_bcm` | float | BCM | yes |
| `gt_pct_filled` | float | % | yes |
| `frl` | float | m | no |
| `live_capacity_bcm` | float | BCM | no |
| `normal_storage_pct` | float | % | yes | rule-curve proxy (ADR-0002) |

### Satellite-derived (from `Observation`; populated on acquisition dates)

| Column | Type | Unit | Null? |
| --- | --- | --- | --- |

### Catchment forcing (from `CatchmentForcing`; native daily)

| --- | --- | --- | --- | --- |

## 3. `ForecastForcing` (Data-Engineering → ML)

Horizon-indexed GFS forecast features. **Point-in-time:** every row is built from the GFS run issued at the most recent cycle **≤ `issue_date`**, read at valid time `issue_date + horizon`. Train and serve use this identical construction (FR-DE-10, FR-ABT-3). Source: GEE `NOAA/GFS0P25` (archive from 2015-07-01 covers the training window).

**Grain:** one row per `(reservoir_id, issue_date, horizon)`.

| Column | Type | Unit | Null? | Notes |
| --- | --- | --- | --- | --- |
| `reservoir_id` | text | — | no | PK |
| `issue_date` | date (IST) | — | no | PK; the as-of date the forecast was knowable |
| `horizon` | int | days | no | PK; 1–14 |
| `forecast_precip` | float | mm/day | yes | catchment-mean GFS precip valid at `issue_date + horizon` |
| `forecast_degree_day_melt` | float | mm/day | yes | from GFS forecast 2 m temperature |
| `gfs_run_cycle` | timestamptz | — | no | the actual GFS run used (e.g. 00/06/12/18Z) — provenance |
| `source_versions` | jsonb | — | no | dataset version |

The forecaster joins `ForecastForcing` to the ABT on `reservoir_id` and `issue_date = date`.
